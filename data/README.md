# Data

Synthetic patient data for developing and testing the fhir-agent platform.

Data is generated by [Synthea](https://github.com/synthetichealth/synthea) — an
open-source synthetic patient generator (Apache 2.0). All patients are entirely
fictional. No real patient data is ever committed to this repository.

---

## How data flows

```
Synthea bundles (data/sample/fhir/)
         │
         │  POST /fhir  (transaction bundle)
         ▼
   fhir-service
         │
         │  HAPI handles: parsing, validation, reference resolution,
         │  search index population, version tracking
         ▼
   Neon PostgreSQL  (or local H2)
```

**Data is always loaded through fhir-service, never written directly to the
database.** HAPI FHIR maintains a normalised internal schema across a dozen
tables. Bypassing fhir-service would leave search indexes unpopulated and
cross-resource references broken.

---

## Getting data into data/sample/fhir/

Two paths — both produce FHIR R4 transaction bundles that `load.py` treats
identically.

### Path 1 — Download pre-generated sample data (fast, no Java needed)

Pulls a ready-to-use patient population from the official
[synthetichealth/synthea-sample-data](https://github.com/synthetichealth/synthea-sample-data)
repository. Good for quick onboarding, CI pipelines, or when you do not need
custom demographics.

```bash
./data/scripts/download_sample.sh
```

### Path 2 — Generate locally with Synthea (full control)

Runs Synthea in the devcontainer (Java 21 is already installed). Downloads the
Synthea JAR (~100MB) on first run and caches it in `data/synthea/`. Subsequent
runs reuse the cached JAR.

```bash
# 100 patients, default demographics (Massachusetts)
./data/scripts/generate.sh -p 100

# 50 female patients with the diabetes module
./data/scripts/generate.sh -p 50 -g F -m diabetes

# 200 patients with diabetes and hypertension
./data/scripts/generate.sh -p 200 -m diabetes -m hypertension

# 100 patients from California
./data/scripts/generate.sh -p 100 -s California
```

Generation configuration is in `data/synthea/synthea.properties`. The most
important setting — `exporter.fhir.transaction_bundle = true` — ensures bundles
are loaded atomically with internal references resolved correctly. Do not change
this to `false`.

## Loading data into fhir-service

Once `data/sample/fhir/` contains bundles (from either path above):

```bash
# Local fhir-service (H2, no API key needed)
FHIR_BASE_URL=http://localhost:8080/fhir python3 data/scripts/load.py

# Deployed stack (Neon via Kong)
FHIR_BASE_URL=http://<kong-ip>:8000/fhir \
FHIR_API_KEY=<your-key> \
python3 data/scripts/load.py
```

The script reports per-bundle success/failure and a final summary:

## Wiping all data and reloading from scratch

Use `expunge.sh` to hard-purge all data from fhir-service. This calls HAPI's
`$expunge` operation, which physically removes all rows from all internal tables
— resources, history, search indexes, version counters, reference links. The
server is left in the same state as a fresh start.

This is **not** a FHIR `DELETE` (which soft-deletes and retains history). It is
a complete wipe.

```bash
# Wipe local server
FHIR_BASE_URL=http://localhost:8080/fhir ./data/scripts/expunge.sh

# Wipe deployed stack
FHIR_BASE_URL=http://<kong-ip>:8000/fhir \
FHIR_API_KEY=<your-key> \
./data/scripts/expunge.sh
```

The script prompts for confirmation before proceeding and verifies the patient
count is 0 after completion.

### Full reload workflow

```bash
# 1. Wipe everything
FHIR_BASE_URL=http://localhost:8080/fhir ./data/scripts/expunge.sh

# 2. Generate a new population (or re-run download_sample.sh)
./data/scripts/generate.sh -p 200 -m diabetes

# 3. Load it
FHIR_BASE_URL=http://localhost:8080/fhir python3 data/scripts/load.py
```

## Verifying the load

### Level 1 — Script output
`load.py` reports per-bundle results and a final resource count. Zero failed
bundles and a non-zero resource count is the primary success signal.

### Level 2 — FHIR API queries

```bash
BASE=http://localhost:8080/fhir

# Patient count
curl -s "${BASE}/Patient?_summary=count" | python3 -m json.tool | grep total

# Medications (should be non-zero — confirms cross-resource references resolved)
curl -s "${BASE}/MedicationRequest?_summary=count" | python3 -m json.tool | grep total

# Allergies
curl -s "${BASE}/AllergyIntolerance?_summary=count" | python3 -m json.tool | grep total

# Conditions
curl -s "${BASE}/Condition?_summary=count" | python3 -m json.tool | grep total
```

Expected after loading 100 Synthea patients (approximate ranges):

| Resource | Typical count |
|---|---|
| Patient | ~100 |
| Encounter | 400–600 |
| Observation | 1,000–3,000 |
| MedicationRequest | 100–300 |
| AllergyIntolerance | 50–150 |
| Condition | 200–500 |
| Immunization | 200–400 |

### Level 3 — Neon spot-check (recommended on first load to a new environment)

Log into the Neon console and run these queries against your database:

```sql
-- Resource distribution — quick population overview
SELECT res_type, COUNT(*) AS count
FROM hfj_resource
WHERE res_deleted_at IS NULL
GROUP BY res_type
ORDER BY count DESC;

-- Reference links — non-zero confirms transaction bundles resolved correctly.
-- If this is 0, MedicationRequest.subject → Patient references are broken.
SELECT COUNT(*) AS reference_links FROM hfj_res_link;

-- Search indexes — non-zero confirms data is queryable via FHIR search.
SELECT COUNT(*) AS string_indexes FROM hfj_spidx_string;
SELECT COUNT(*) AS token_indexes  FROM hfj_spidx_token;
SELECT COUNT(*) AS date_indexes   FROM hfj_spidx_date;

-- After expunge — all of these must return 0
SELECT COUNT(*) FROM hfj_resource;
SELECT COUNT(*) FROM hfj_res_link;
```

The Neon check is optional for routine reloads. Run it once when setting up a
new environment to confirm the full storage pipeline is working correctly.

## Who uses this data

| Audience | How they use it |
|---|---|
| **Hat 1 — Platform engineers** | Load large populations to stress-test fhir-service performance, indexing, and schema migrations. Use `load.py` directly against a local server. |
| **Hat 2 — Clinical developers** | Load a realistic patient population to develop and test clinical workflows (triage, prior auth) via `FHIRClient`. |
| **MCP agent** | Queries the loaded data through `FHIRClient` during walking skeleton development. |

Both audiences use the same scripts and the same data. The loading path is
always through fhir-service — never directly to the database.

## What is not committed

```
data/sample/fhir/       # Generated bundles — can be hundreds of MB
data/synthea/*.jar      # Synthea JAR — downloaded on demand by generate.sh
data/synthea/output/    # Raw Synthea output before moving to sample/
data/scratch/           # Ad-hoc experiments
```

See `data/.gitignore` for the full exclusion list.
